Mapping Findity’s headless expense platform against the Centime Expense Management product specification. Can we build what we designed?

February 2026 · Product & Engineering · Internal

The Architecture Question

What is the boundary between our layer and Findity’s?

Our spec calls for a policy-driven expense platform built on Findity’s white-label framework. But the spec was written before we had API documentation. Now we have it. The question: does the API surface match what we need? Where does Findity end and Centime begin?

The answer is clear — and it’s good news.

Platform Architecture

Four APIs, headless by design

Expense

Expense API

User-facing: create/edit expenses, mileage, per-diem, reports, approvals, receipt management, route calculation.

Admin

Admin API

Org setup: users, roles, departments, dimensions (cost centers/projects), approval chains, policies, custom fields.

Connect

Connect API

Inbound card transactions, outbound GL export vouchers, webhooks for event-driven integration.

Onboard

Onboarding API

Subset of Admin for customer provisioning — automates multi-tenant setup.

Engine Capabilities

What Findity provides out of the box

01

Three expense types

Standard expenses (ReceiptVerification), mileage (CarSpecification), per-diem (SubsistenceAllowance). All three in our spec are native.

02

Card transaction ingestion

POST /card-transactions. Real-time events: auth, capture, refund. Auto-creates expenses. Push notifications to users.

03

Three liability models

PRIVATE (personal, reimbursable), CORPORATE (company invoice), DEBET (direct charge). Maps to our personal-vs-corporate split.

04

Dimensions & custom fields

Unlimited dimensions for GL codes, cost centers, projects. Custom fields extend any expense type. Dynamic fields endpoint.

05

Approval chains

Dimension-based multi-level approval with sort-order sequencing. Approvers per dimension value.

06

Export & vouchers

Connect API exports structured voucher data for GL journal entry. Status management for processing pipeline.

Spec Section 03

Policy engine: our biggest original build

Findity Provides

Policy document CRUD. Categories and expense types. Custom field definitions and value lists. Dimensions for cost center / project tagging.

We Build

Graduated card mandate (4 levels). Per-category spend limits and caps. Soft/hard/block violation handling. Policy hierarchy evaluation. Policy visibility at every stage. All enforcement UX.

The policy engine is the single largest piece of original business logic. Findity gives us the building blocks. The rules are ours.

Spec Section 04

T&E expenses: strong native coverage

Card transactions — POST with real-time events, auto-expense creation, push notifications Native
Three expense types — standard, mileage, per-diem Native
Receipt/content management — upload, retrieve, OCR scanning Native*
MCC auto-categorization — merchant codes mapped to categories Native
Mileage route calculation — directions API with map visualization Native
Card feed integration — US Bank / FNBO → Findity push We Build
IRS rates, GSA per-diem, commute offsets We Configure
Email forwarding, match scoring, receipt affidavit We Build

*OCR: Findity claims ‘advanced scanning’ — need to confirm structured data extraction via API.

Spec Section 06

Approval workflows: Findity executes, we route

Findity Executes

Dimension-based approval chains. Multi-level with sort-order. Approve/reject actions on reports. Approver assignment per dimension value.

We Route

Amount-threshold rules. Category-based routing. Violation-triggered extra approvers. Auto-approval engine. Delegation with date ranges. Aging escalation. Batch approval. Return for correction. Approve with modification.

Findity’s approval infrastructure is the execution layer. Our routing engine decides who approves what and when — then uses Findity’s chain to make it happen.

Spec Sections 07–08

Finance ops and integrations: mostly our build

Findity: Export

Voucher data via Connect API

Structured for GL journal entry. Status management for processing pipeline.

Findity: Reports

Report status & financial data

Report status tracking. Report counters. Card transaction financial data (amount, currency, MCC, liability).

We Build: Finance

Full finance operations

5-status reconciliation workspace. Period close with automated reminders. Reimbursement batches + ACH. GL auto-coding. Split coding. All dashboards.

We Build: Integrations

All connectors

ERP connectors (QBO, NetSuite, Sage, Dynamics). HRIS sync. SSO. Card feed push. Slack/Teams. Calendar.

Spec Section 05

Basic purchasing: 100% our module

Findity is an expense platform, not a procurement platform. Purchase requests, PO generation, three-way matching, and pre-approval routing are entirely ours.

The integration point is post-purchase: when a card transaction creates a Findity expense, we match it to our approved PO, inherit the GL coding, and close the three-way loop.

Open Questions

Eight questions for the Findity team

OCR

Structured receipt data?

Does the API return structured receipt data (amount, date, vendor, tax)? Or just store images?

Amount Routing

Threshold-based approvals?

Is amount-threshold approval routing native? Docs show dimension-based only.

Split Coding

Multi-dimension coding?

Can one expense be coded to multiple cost centers natively? Critical for our GL requirements.

US Market

US tax & rate config?

GSA per-diem rates, IRS mileage rates, US tax rules — available or do we configure from scratch?

API Stability

Production-grade SLA?

Docs note ‘Beta-stage.’ What is the SLA, versioning policy, and breaking change commitment?

ERP Connectors

Existing integrations?

Existing QBO, NetSuite, Sage, Dynamics connectors? Or do we build from voucher exports?

Card Feed

US bank card format?

Exact payload spec for POST /card-transactions? Do US Bank / FNBO feeds map cleanly?

Mobile

White-label app as MVP?

Can we use their mobile app as V1.0 MVP and replace with custom later?

The Clear Line

Findity is the engine. We are the experience.

Findity Owns

Expense data storage & processing. Card transaction ingestion. Expense types (standard/mileage/per-diem). Approval chain execution. Receipt/content storage. Voucher export for GL. User & org data model. API infrastructure & uptime.

Centime Owns

Every pixel the user sees. Policy enforcement engine. Approval routing logic. All finance workflows. Basic purchasing module. ERP/HRIS/SSO connectors. Mobile applications. Reporting & analytics. Card feed integration. The business.

Next Steps

Three things before engineering starts

  • 1 Schedule a technical discovery call with Findity’s API team. Walk through the eight open questions — especially OCR, amount routing, and US market configuration.
  • 2 Request sandbox access and build a proof-of-concept: push a card transaction, create an expense, run it through an approval chain, export a voucher. Validate the end-to-end flow.
  • 3 Confirm API stability commitment and SLA. The ‘Beta-stage’ label needs to become a production-grade contract before we build on it.
The API fits. The architecture is right. The open questions are answerable. Let’s get the answers.
