00 — What We Specified

The product we designed

Before mapping the Findity API, a summary of what we set out to build. The full product specification is available as a scrollable document and a slide deck.

Centime Expense Management is a policy-driven expense and purchasing platform for US mid-market finance teams (200–2,000 employees), built on Findity’s white-label T&E engine. We own the UX and workflows; Findity provides the expense processing engine. The platform enforces corporate card usage as the default, provides configurable policy guardrails, and delivers a streamlined experience across three personas: employee (submitter), manager (approver), and finance (controller).

Design Principles

Corporate Card First

US Bank (primary) and FNBO (backup). Every UX nudge favors our card. Card-agnostic acceptance, but pricing rewards our programs.

Policy at the Center

Rules shape the interface, the options available, and the approval routing. Not bolted on after the fact.

Built on Findity

Findity provides the expense engine. We bring UX, workflows, card partnerships, and Centime platform integration.

Accountability Without Friction

Capture what finance needs for audit and GL coding without making employees hate the process.

Feature Summary

  • Policy engine: Four corporate card mandate levels (Encouraged, Preferred, Required, Mandatory). Policy hierarchy (company → department → role → project). Per-category spend limits, receipt thresholds, per-diem and mileage rules. Three violation responses: soft warning, hard warning, block.
  • T&E expense management: Corporate card transaction auto-ingestion with MCC categorization. Personal card exception path with justification. Mobile receipt capture with OCR. Mileage (IRS rates, Google Maps, commute offsets). Per-diem (GSA rates, custom rates, partial-day). Trip-based and period-based expense reports.
  • Purchasing: Full purchase lifecycle — request, routing, approval, virtual card creation, tracking, reissue, cancellation. PO-based and non-PO paths. Virtual card as control mechanism (vendor-locked, amount-limited). Invoices flow to Centime AP for payment.
  • Approval workflows: Multi-dimensional routing by amount, category, department, and violation status. Auto-approval for clean, in-policy expenses. Delegation with date ranges. Aging escalation. Batch approval. Return for correction and approve with modification.
  • Finance operations: Five-status card reconciliation workspace. Period close with automated reminders and manager escalation. Reimbursement batch processing via ACH or payroll. GL auto-coding (MCC-to-GL mapping), split coding, GL validation. Full reporting suite: compliance scorecard, card ratio, unreconciled aging, spend analysis, cycle time, duplicate detection.
  • Integrations: ERP/GL push (QuickBooks, NetSuite, Sage Intacct, Dynamics). HRIS sync (BambooHR, ADP, Workday). SSO (Okta, Azure AD, Google). Visa/MC card feeds from US Bank and FNBO. Slack/Teams approval notifications. Calendar expense association.
  • Mobile: Full employee workflow (receipt capture, quick-submit, mileage, status tracking, purchase requests). Full manager workflow (swipe approve/reject, policy flags, batch approve, delegation). Finance operations are desktop-only.
  • Security & compliance: Five RBAC roles. Tamper-evident audit trail on every action. SOC 2 Type II. PCI compliance via tokenized card feeds. Encrypted receipt storage with configurable retention.

The Question This Document Answers

Does the Findity API surface support what we specified? Where does Findity end and Centime begin? What do we need to build ourselves, and what gaps require resolution before engineering starts?

01 — Findity Architecture

Findity’s headless expense engine

Findity is a headless expense management platform. It exposes four APIs that collectively cover expense creation, organizational administration, card transaction ingestion, and customer onboarding.

The architecture is REST-based, with OAuth 2.0 authentication and JSON request/response throughout. Findity’s positioning statement captures the relationship precisely: “We provide the engine; you define the user experience.”

This matches our spec’s architecture exactly. Findity handles expense processing — the data model, the storage, the calculation logic, the approval chain execution. We own the UX, the workflows, the policy enforcement, and all persona-specific experiences.

Partnership framing. This assessment maps the current Findity API surface against our specification. Where gaps exist, we expect Findity to enhance their platform to close them. This is a partnership, not a one-way API consumption. Findity has an incentive to build what we need — our deployment drives meaningful volume through their platform. Capabilities listed as gaps are questions for the partnership discussion, not reasons to build our own.

Expense API

User-facing operations: create/edit expenses, mileage, per-diem, reports, approvals, content/receipt management, route calculation.

Admin API

Org setup: users, roles, units/departments, dimensions/cost centers, approval chains, policies, categories, custom fields.

Connect API

Inbound card transactions, outbound GL export vouchers, webhooks for event-driven integration.

Onboarding API

Subset of Admin for customer provisioning — streamlined endpoints for initial organization and user setup.

Note on Stability

The API documentation notes that the platform is currently in “Beta-stage.” We need to understand the stability commitment, SLA guarantees, versioning policy, and breaking change notification process before committing to production integration.

02 — Mapping to the Spec: Overview & Personas

Organizational hierarchy and user roles

Spec sections 01–02. Findity’s organizational hierarchy — Client App → Partner → Organization → Units — maps directly to our multi-tenant, multi-customer bank-channel model. The bank is the Partner. Each bank customer is an Organization. Departments within the customer are Units.

Findity supports five native user roles: End User, Approver, Org Admin, Partner Admin, and Accounting Consultant. These map to our spec’s personas: Employee, Manager, Finance Admin, Finance Viewer, and System Admin.

User lifecycle management is fully programmatic via the Admin API — create, update, and delete operations enable HRIS sync. External ID support throughout the data model enables SSO integration. Because the platform is headless, all persona-specific UX is entirely ours to design. Findity does not constrain the employee, manager, or finance experience in any way.

Findity Provides

  • Multi-level org hierarchy (Client App → Partner → Org → Units)
  • Five native user roles with RBAC
  • External ID support for SSO mapping
  • Programmatic user provisioning (CRUD via Admin API)

We Build

  • All persona-specific UX (employee, manager, finance admin)

03 — Mapping to the Spec: Policy Engine

The biggest “we build” item

Spec section 03. Findity provides the data substrate for policy configuration — categories, dimensions, custom fields. But the enforcement logic, the violation handling, and the graduated card mandate are entirely ours to build.

Findity has policy documents (CRUD per organization), categories with expense types, custom field definitions with value lists, and unlimited configurable dimensions. These are the building blocks.

Findity does not have our graduated card mandate levels (Encouraged / Preferred / Required / Mandatory), per-category spend limits, soft/hard/block violation handling, policy visibility at pre-trip/submission/rejection, or policy hierarchy evaluation (company → department → role → project).

04 — Mapping to the Spec: T&E Expense Management

Expense types, card transactions, and receipts

Spec section 04. Findity natively supports all three expense types in our specification: ReceiptVerification (standard expense), CarSpecification (mileage), and SubsistenceAllowance (per-diem).

Card transaction ingestion

The Connect API accepts card transactions via POST /card-transactions. Supported event types include AUTHORIZATION_ACCEPTED, AUTHORIZATION_REJECTED, AUTHORIZATION_REVERSAL, CAPTURE, and REFUND. Transactions auto-create expenses and trigger push notifications. Three liability types map to our card models: PRIVATE (personal card, reimbursable), CORPORATE (company card, invoice-matched), and DEBET (direct corporate charge).

Receipt and content management

Upload, retrieve, delete, and get-as-image endpoints are available. The platform claims “advanced scanning” with OCR. Open question: Does the API return structured OCR data (amount, date, vendor, tax) or just store images?

Mileage and per-diem

Route calculation with map visualization, car type configurations, distance-based reimbursement, and vehicle type selection are built in. Open questions: Are IRS standard rates, Google Maps integration, commute offsets, and recurring route templates native or do we configure them? For per-diem, are GSA rates pre-loaded for US cities, or do we configure?

Dynamic fields

The fields-format endpoints return which fields are relevant for each expense type per organization — this handles per-org customization dynamically. This is the recommended approach for form rendering.

05 — Mapping to the Spec: Purchasing

Full lifecycle is ours; invoices hand off to AP

Spec section 05. The purchasing module manages the complete lifecycle: request, routing, approval, virtual card creation, tracking, reissue, and cancellation. Purchases may or may not be PO-based. When a purchase results in an invoice, that invoice is processed and paid through Centime AP.

Findity has no purchase request concept. Its role is limited to the card transaction layer: when the employee uses the virtual card, the transaction flows into Findity via the Connect API. Our layer matches the Findity expense to the approved purchase request and inherits GL coding.

The virtual card is the control mechanism. Instead of giving employees access to a shared corporate card, the system issues a vendor-locked, amount-limited virtual card for each approved purchase. This gives finance real-time visibility and control — and the card can be reissued, cancelled, or tracked at any point.

The purchasing module owns the full purchase lifecycle. Findity processes the card transaction. Centime AP processes any resulting invoices. Three systems, clean boundaries: purchasing (request → virtual card → tracking), Findity (card transaction), AP (invoice → payment).

06 — Mapping to the Spec: Approval Workflows

Routing logic is ours; execution is Findity’s

Spec section 06. Findity has dimension-based approval chains. Approvers are assigned per dimension value (e.g., cost center 100 → Manager X). Multi-level chains are supported with sort-order sequencing. The Expense API supports approve/reject on reports via PUT /approvals/report/{reportId}.

What Findity does not explicitly support — or what remains unclear from the documentation — includes amount-threshold routing ($500 → manager, $5K → VP), category-based routing, violation-triggered extra approvers, auto-approval, delegation with date ranges, aging escalation, batch approval, return-for-correction, and approve-with-modification.

Our layer builds the routing rules engine that evaluates amount, category, department, and violation status, then uses Findity’s approval chain as the execution mechanism.

07 — Mapping to the Spec: Finance Operations

Export data from Findity; workflows from us

Spec section 07. Findity’s Connect API exports voucher data for GL/accounting integration. The Admin API provides report status management. The financial data substrate is there. The operational workflows are ours.

The Connect API provides export endpoints with status management — list, get, process, and get content. Card transaction data includes all the financial fields needed for reconciliation: billing amount/currency, merchant amount/currency, MCC, liability type, and invoice number for corporate cards. Report status management via the Admin API supports list, get, and update operations with report counters.

What we build entirely: the 5-status reconciliation workspace, period close workflow with T-5/T-3/T-1 reminders, reimbursement batch processing and ACH payment, GL auto-coding (MCC-to-GL mapping rules), split coding across cost centers, GL validation, period assignment for accrual accounting, and all finance dashboards and reports.

09 — Mapping to the Spec: Mobile, Security & Release

Infrastructure and phased rollout

Spec sections 09–11. Findity provides the API infrastructure and data security layer. We build the client applications, the SSO integration, and maintain our phased release schedule.

Mobile

Findity has white-label mobile apps available, but we’re building custom UX. The Expense API is fully mobile-capable — all endpoints work from any client. The route/directions API supports GPS mileage capture. Push notifications are available for card transaction events.

Security

Findity handles data storage, PCI compliance (tokenized card data), and provides RBAC with five role types. We own SSO integration, additional access controls, and the audit trail UX.

Release schedule

No changes to our phased approach. Findity integration is the V1.0 foundation. Each subsequent release layers our UX and business logic on top.

  • V1.0 MVP — Findity platform integration, card feed (Findity-provided), basic expense CRUD
  • V1.1 — Policy engine (our build, uses Findity categories/dimensions)
  • V1.2 — Finance ops (our build, uses Findity exports/reports)
  • V1.3 — Personal cards (Findity PRIVATE liability type)
  • V1.5 — Mobile (our app, Findity Expense API)
  • V2.0 — Purchasing (virtual card lifecycle, invoices → Centime AP) + ERP connectors (Findity export → our transform)
10 — Gaps and Open Questions

Eight questions for Findity

These questions must be answered during technical discovery. Each one has the potential to shift scope between “Findity handles” and “we build.” None of them require architectural changes to our spec — they determine implementation depth, not design direction.

  1. OCR capability. Does the Expense API return structured receipt data (amount, date, vendor, tax) or just store images? This determines whether we build our own OCR pipeline.
  2. Amount-threshold approval. Is amount-based routing native or do we build it? The docs show dimension-based routing only.
  3. Expense splitting. Can one expense be coded to multiple cost centers/dimensions natively? Critical for split coding.
  4. US market configuration. Are GSA per-diem rates, IRS mileage rates, and US tax rules available? Findity’s strongest markets are Nordic/EU.
  5. API stability. The docs note “Beta-stage.” What is the SLA, versioning policy, and breaking change commitment for production use?
  6. Pre-built ERP connectors. Do they have existing QuickBooks, NetSuite, Sage Intacct, or Dynamics connectors we can leverage, or do we build from voucher exports?
  7. Card feed integration. What is the timeline and scope for Findity to build US Bank and FNBO card feed integrations? What data format and onboarding steps are required from the card issuers?
  8. White-label mobile. Can we use their white-label mobile app as V1.0 and replace with custom later, or is it all-or-nothing?
11 — The Architecture Boundary

Findity is the engine. We are the experience.

The Clear Line

Findity is the expense processing engine. We are the experience, enforcement, and integration layer.

Findity owns:

  • Expense data storage and processing
  • Card transaction ingestion and auto-expense creation
  • Expense type management (standard/mileage/per-diem)
  • Approval chain execution (dimension-based)
  • Content/receipt storage
  • Voucher export for GL
  • User and org data model
  • API infrastructure and uptime
  • US Bank/FNBO card feed integration

We own:

  • Every pixel the user sees — all three personas
  • Policy enforcement engine (4 mandate levels, spend limits, violations)
  • Approval routing logic (amount, category, department, violation triggers)
  • Auto-approval, delegation, escalation
  • All finance workflows (reconciliation, reimbursement, period close)
  • Purchasing module (request → virtual card → tracking; invoices → Centime AP)
  • ERP/GL connectors (QBO, NetSuite, Sage Intacct, Dynamics)
  • HRIS sync, SSO, mobile applications
  • Reporting and analytics

This assessment confirms that Findity’s headless architecture is the right foundation for Centime Expense Management. The API surface covers the expense processing engine we need. The open questions are answerable — they require technical discovery conversations with Findity’s team, not architectural changes to our spec. The critical path is: confirm OCR capability, validate US market support, and establish the API stability commitment.
